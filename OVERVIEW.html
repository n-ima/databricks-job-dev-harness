Why

こんな迷い、ありませんか？

AIコーディングは速い。しかしDatabricks Jobは無人で本番データに触り続ける。速さと安全さを両立する「型」がないと、結局は人の注意力に頼ることになる。

ハーネスなしで進めると

  • 設計書の粒度がバラバラで、実装のたびに手戻りする
  • 誰が実装しても品質にムラが出る
  • 本番デプロイの確認が「気をつける」任せ
  • 緊急修正でコードだけ直し、設計書との食い違いに気づかない
  • チャットが長くなるほど応答が遅く・的外れになる

このハーネスを使うと

  • IRRが「実装可能な設計書か」を機械的にGO / NO-GOで判定
  • 独立レビュー（design-critic / reviewer）が思い込みを検出
  • Hooks + GitHub Environments + 権限分離の三重で本番を保護
  • 緊急修正は許容しつつ、設計との乖離を期限つきで追跡
  • セッションを分けるタイミングをエージェント自身が提案
How it flows

全体の流れ — 人が関わるのは要所だけ

設計書チェックとproductionリリースの承認だけが人の判断ポイント。実装からテストまではノンストップの全自動区間として設計している。

設計書チェック（IRR）とproduction承認だけが人の判断ポイント。それ以外はノンストップで進む。

Architecture

3層構造でエージェントを迷わせない

「誰が」「どうやるか」「機械的にどう守るか」を分離しているから、各エージェントは薄いまま保たれ、無関係な情報でコンテキストを圧迫しません。

Agents（誰が）
フェーズごとの人格・ツール権限・引き継ぎ先。design-critic / task-worker / reviewer は独立コンテキストのサブエージェント。
.github/agents/*.agent.md
Skills（どうやるか）
IRR判定・テスト設計・デプロイ手順などの部品。使う場面が来るまで本文を読み込まない「段階的開示」でトークンを節約。
.github/skills/*/SKILL.md
Hooks（機械的にどう守るか）
モデルを介さないシェル実行。prod操作をask確認、ハーネス設定の無断編集をdeny、ゲート状態を毎セッション自動注入。
.github/hooks/*.json
Design considerations

考え抜かれた10の工夫

機能を並べただけでは終わらない。実際に運用したときに壊れないための工夫を積み重ねている。

01

設計書チェックが最初の関門

IRRがGO / CONDITIONAL GO / NO-GOを判定。機械可読マーカーとしてセッション開始のたびに自動注入され、NO-GOの間は実装コードに触れない。

02

独立レビューでバイアス除去

design-critic（設計書）とreviewer（コード）は実装者と地続きにしない、まっさらなコンテキストで起動。自己承認バイアスを構造的に防ぐ。

03

本番保護の三重化

Hooks（staging/prod操作をask）× GitHub Environments（required reviewers）× Databricks権限分離（devに書き込み権限なし）。1つが効かなくても止まる。

04

冪等性を「証明」してから合格

同じJobを同じパラメータで2回実行し、結果が変わらないことを確認して初めてテスト合格。単体テストのassertが通るだけでは終わらせない。

05

サブエージェントでコンテキストを分離

1タスク＝1回の独立コンテキスト実行。長い会話ほど精度が落ちる"context rot"を避けつつ、コーディネーター自身の会話は軽いまま。

06

差分駆動の保守

バグ修正は再現テストだけで完結し、全体の再チェックはしない。影響範囲はトレーサビリティ表から機械的に辿る。トークンを無駄にしない。

07

使うほど賢くなる成長ループ

試行錯誤で確立した実行方法・振り返りの改善提案が、次のプロジェクトの初期状態に資産化される。

08

エージェントが使い方まで案内

次の一手・新チャットに切り替えるタイミング・正しい入口を、エージェント自身が提案する。操作手順の暗記を前提にしない。

09

コストを見て使い分けるモデル選択

高頻度・機械的な作業は軽量モデル、設計判断や独立レビューだけ高性能モデル。Skillの段階的開示とあわせてトークン消費を抑える。

10

操作はCLI、知識はAI Dev Kit

Databricks CLI + Asset Bundlesを操作の正とし（Hooksで検査できるため）、AI Dev Kit/MCPは構文知識とワークスペース調査に専念させる役割分担。

Scope

向いているプロジェクト・向いていないプロジェクト

固定費（設計登録＋IRRで数時間）と1タスクごとの儀式コストは、無人で本番稼働するJobで初めて回収できる。対象外に使うと明確に過剰。

使うべき場面

  • 本番スケジュールで無人実行されるJob
  • 複数人・長期保守が前提の開発
  • 監査要件・証跡が求められる環境

使わなくてよい場面

  • 使い捨ての分析ノートブック・アドホック処理
  • PoC・技術検証（本番化が決まってから載せ替える）
  • 1人で完結し、記録を残す必要のない小さな作業
