# 実装準備レビュー（IRR）

目的は設計レビューのやり直しではなく、**「設計レビュー通過」と「この方式で安全に
実装・試験・運用できる」のギャップを埋める**こと。業務設計そのものは変更しない。

## 読む範囲

最初に `docs/01-design/design-index.md` と `docs/01-design/irr.md`（無ければ
`irr_template.md` からコピー）だけを読む。索引からIRRに必要な承認済み設計だけを
追加で読む。要件文書が「なし」なら要求しない。

## 手順

1. 基本設計と全機能・共通機能の詳細設計が索引に登録され、版・承認日・正本が
   一意であることを確認する。`TODO`・未承認の行が残る場合、その設計IDは判定対象外
   （= その分のスコープはNO-GO扱い）とする。
2. `design-critic` サブエージェントの品質レビュー結果（呼び出し元が実施済み）の
   BLOCKER/MAJORを未解決事項表へ転記する。
3. `irr.md` の必須チェックを1項目ずつ、設計書の該当記述と突き合わせて確認する。
   主要カテゴリ:
   - **要求・境界**: 入出力・データ型・主キー・重複/欠損/遅延データ、正常/異常/再実行の
     期待結果、受け入れ条件のID化、対象外の明示
   - **Databricks適合性**: JobのタスクDAG・依存・パラメータ・タイムアウト・再試行、
     compute選択（classic/serverless・Runtime・node type・policy）—
     **environment.mdの記載を鵜呑みにせず、実機制約（サーバレス専用ワークスペース、
     利用可能なnode type/policy）と一致するか実ワークスペースで確認する**、
     dev/staging/prodのcatalog/schema分離、Unity Catalogの所有者・実行主体・最小権限、
     データ量・SLA/SLO・コスト上限・スケジュール・タイムゾーン、
     冪等性・チェックポイント・部分失敗・バックフィル方式
   - **テスト可能性**: 業務ルールのSpark非依存分離可否、合成/匿名化テストデータ、
     結合テスト用schema・権限、データ品質合格基準の数値化
   - **運用・リリース**: 監視指標・通知先・Runbook・一次対応、本番承認者・変更可能時間帯、
     rollback/roll-forward基準、データ復旧・保持・監査
   - **AI利用**: 組織ポリシー上の許可、AIへ送信禁止の情報区分、生成コードの人的レビュー、
     AI Dev Kit/MCP/skillsのバージョン固定
4. 確認できない項目は「未確認」とし、**推測で埋めない**。不足ごとに重大度・担当・期限・
   客観的な解決条件を未解決事項表に記録する。
5. 判定する。
   - `GO`: 実装上の重要な曖昧さがない。
   - `CONDITIONAL GO`: 安全に実装できる範囲内で、担当と期限のある条件だけが残る
     （例: 通知先メールアドレスが未定だが実装には影響しない）。
   - `NO-GO`: 推測実装になる、または許容できないセキュリティ/データ/運用リスクが発生する
     （例: 再実行時の仕様が未定義、prodの権限モデルが未定）。
6. 判定・blocker・条件・**最初に実装すべき小さな単位**（薄い縦切りの1機能）を報告し、
   人の承認を求める。このスキルでは業務コードを実装しない。

## 再判定（改修・保守時も同じ）

実装中またはリリース後に設計変更・前提の崩れが判明した場合、変更された設計IDに
関する項目だけを再確認して `irr.md` を更新する（全項目のやり直しはしない）。

- **設計変更を伴わないバグ修正**（実装が設計とずれていた）はIRRの再判定対象外。
  そのまま実装フェーズで再現テスト→修正を行う。
- `design-index.md` に「実装乖離あり」の設計IDが残っている場合、その乖離を
  期限つきの未解決事項として扱い、解消されるまで判定は `CONDITIONAL GO` を上限とする。
