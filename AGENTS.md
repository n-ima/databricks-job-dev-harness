# AGENTS.md — このリポジトリで動くすべてのAIエージェント向け共通指示

このファイルは GitHub Copilot（VS Code）を主対象としつつ、Copilot coding agent（cloud）、
Claude Code など `AGENTS.md` 標準に対応する他のエージェントでも同じ運用ルールが効くように
するための、**唯一の正**となる指示ファイルです
（`.github/copilot-instructions.md` と重複させず、本ファイルに一本化しています）。

回答と文書は日本語で記述する。コード・API・コマンドの識別子は英語を維持する。

## このリポジトリについて

Databricks Job（Lakeflow Jobs）を対象にした **開発ハーネス（雛形テンプレート）** です。
要件定義・業務設計はこのハーネスの **外** で完了している前提で、入力は「承認済みの設計書」。
「設計書チェック → 実装計画 → 実装 → テスト → リリース」をオーケストレーションします。

## フェーズとゲート：フェーズごとに「人の関わり方」が違う

1. **設計書登録・チェック**（`docs/01-design/`）— **人が最も関わるフェーズ**。
   設計書を配置・索引登録し、`environment.md`（ワークスペース・カタログ・自動化境界）を
   仮置きせず具体的に埋める。設計書に不足がある場合や改修で設計を変える場合は
   `/08-design-revise`（design-gateの改訂モード）で、1決定ずつ人間と合意しながら
   設計書を改訂できる（AIが黙って推測で補完することだけが禁止）。
   `design-critic` サブエージェントの品質チェックと
   実装準備レビュー（IRR: `docs/01-design/irr.md`）で `GO` / `CONDITIONAL GO` / `NO-GO` を
   判定し、人が承認する。**IRRが未判定または `NO-GO` の間、プロダクションコード
   （`src/`, `resources/`, `databricks.yml`）を実装・変更しない。**
2. **実装計画**（`docs/02-implementation/`）・**3. 実装** — **全自動区間**。
   IRR承認後、人の逐一確認なしでノンストップに進む。実装は `task-worker` サブエージェントに
   1タスクずつ委譲する。人が関わるのは真のブロッカー（設計だけでは判断できない事実が
   必要なとき）と、Hooksが機械的に確認を入れる操作のときだけ。
4. **テスト**（`docs/03-test/`）— **全自動区間**。ローカル単体テスト → wheelビルド →
   `bundle validate` → **devターゲットへのdeploy → Job実行 → 結合テスト** まで自動で行う。
   完了後・リリース前に、実装した本人とは別コンテキストの `reviewer` サブエージェントが
   独立レビュー（正しさ・Databricks固有リスク・セキュリティ）を行い、問題があれば差し戻す。
5. **リリース**（`docs/04-release/`）— staging検証は自動、**productionデプロイの承認は必ず人**。
   AIは自己承認しない。`git push`・タグ付け・staging/prodへの`bundle deploy`はHooksでも
   機械的に確認が入る。

- 各フェーズはこの順に進める。フェーズを飛ばさない。
- 成果物は必ず `docs/` 配下に Markdown で残す。会話内だけで完結させない。
- 現在のゲート状態は `docs/00-overview/progress.md` の `GATE_STATUS` ブロックが正
  （判定手順は `.github/skills/gate-check/SKILL.md` を参照）。

## Databricks開発の絶対ルール

- **Databricksリソースの正は Asset Bundles**（`databricks.yml` / `resources/*.yml`）。
  Job・権限・スケジュールをUIで恒久変更しない。UIは確認・監視にのみ使う。
- 業務ロジックはノートブックではなく `src/` のwheelパッケージに置き、
  可能な限りSpark非依存の純粋関数にする。
- 環境差分（catalog / schema / 通知先 / service principal 等）はBundle変数と
  ターゲット定義に置く。コードへの直書き禁止。
- シークレット・トークン・個人情報・顧客データ・実データの行を、プロンプト・コード・
  テスト・ログ・コミットのいずれにも含めない。失敗調査はマスキング済みの
  schema・error class・run IDで行う（`docs/01-design/environment.md` の機密区分に従う）。
- Spark非依存のテストは `tests/unit/`、ワークスペース・デプロイ済みリソースが必要な
  テストは `tests/integration/`（`integration` marker付き）に分離する。
- 設計にない業務仕様をAIの推測で補完しない。`TODO`・未承認・版不明の設計書を
  実装根拠として使わない。不足はIRRの未解決事項として記録し、人に確認する。
- 実装完了の最低条件: `uv run task check`（format/lint/型/単体テスト）と
  `uv run task build` の成功。認証済みなら `databricks bundle validate -t dev` も実行する。

## エージェント構成（VS Code Copilot）

- `.github/agents/*.agent.md` — フェーズごとの専属エージェント。
  `orchestrator` → `design-gate` → `implement` → `test` → `release` の順で
  `handoffs` により引き継ぐ。
- `.github/agents/design-critic.agent.md` — **サブエージェント**。`design-gate` がIRR判定前に
  `runSubagent` で呼び出す、設計書の独立レビュー担当（読み取り専用）。
  上流の欠陥は下流ほど修正コストが大きいため、コードレビューより先にここへ独立レビューを置く。
- `.github/agents/task-worker.agent.md` — **サブエージェント**。`implement` がタスクを
  1つ実装するたびに呼び出す。実装ループ全体を1つの長い会話にしないための
  コンテキストロット対策。
- `.github/agents/reviewer.agent.md` — **サブエージェント**。`test` がリリース前に呼び出す、
  実装とは独立したコンテキストのコード/変更レビュー担当（読み取り専用）。
- `.github/skills/*/SKILL.md` — 各フェーズで使う手順・チェックリストの部品
  （ゲート判定、IRR、テストケース設計、デプロイ、環境セットアップ、セキュリティレビュー、
  振り返り、Skill作成）。Databricks公式のAgent Skills（`databricks aitools` 経由）を
  導入した場合は、製品構文・一般的手順の根拠として使い、案件固有の設計より優先しない。
- `.github/prompts/*.prompt.md` — フェーズを進めるスラッシュコマンド。各プロンプトは
  frontmatterの `agent:` で専属エージェントにバインドしてある。
- `.github/hooks/` — ゲート・セキュリティ・本番保護を補強する機械的なフック。
  Preview機能のため過信せず、本ファイルの指示レベルのルールと併用する。

## ナビゲーション責務（全エージェント共通のメタガイダンス）

このハーネスは操作を覚えていないユーザーでも迷わないよう、**エージェント自身が
使い方を案内する**。全エージェントは次を守る（詳細は `harness-guide` スキル）。

- **応答の最後に次の一手を1行で案内する**: 次に実行すべきプロンプト（またはハンドオフ）と、
  「新しいチャットにすべきか・続けてよいか」を、USAGE.mdのセッション表に従って添える
  （例:「次は新しいチャットで /02-design-check を実行してください。改訂の文脈を
  チェックに持ち込まないためです」）。
- **フェーズ違いの依頼は入口から案内する**: 現在のフェーズ・エージェントに合わない依頼
  （実装セッション中の設計変更相談など）を受けたら、作業を始める前に正しいプロンプトと
  セッションを案内する。
- **セッションの切り替えを自分から提案する**: 完了タスク10個超・差し戻し2往復超・
  会話の劣化を感じたら、区切りのよいところで新チャットへの切り替えを提案する
  （正の状態は `docs/` にあるため何も失われないことを添える）。
- **コストに影響する選択は理由つきで提案する**: 強いモデルへの切り替え・並列レビュー等は
  勝手に増やさず、費用対効果の理由を添えて提案する。
- 使い方そのものの質問には `/09-harness-help`（harness-guideスキル）で答える。

## サブエージェント活用方針

- 各エージェントの `agents` frontmatterで、呼び出せるサブエージェントを明示的に絞る
  （最小権限。既定は `[]` = サブエージェント呼び出し不可）。
- 既定で使う経路は次の3つのみ。
  - `design-gate → design-critic`（IRR判定前に1回）: 設計書の品質・実装可能性の独立レビュー
  - `implement → task-worker`（タスク数だけ）: コンテキストロット対策の実装分離
  - `test → reviewer`（1回）: コードの正しさ・Databricksリスク・セキュリティの独立レビュー
- 並列の多視点レビューのような追加呼び出しは、コストに対して効果が見合うかの判断が
  要るため、ユーザーが明示的に要求した場合のみ増やす。

## セキュリティガードレール

- 各エージェントの `tools` は必要最小限にする（`design-gate` に `execute` を持たせない等）。
- ハーネス自体の設定ファイル（`.github/agents/`, `.github/hooks/`, `AGENTS.md`,
  `plugin.json`, `.vscode/settings.json`）は、プロンプトインジェクション等による
  自己権限昇格・ガードレール解除を防ぐため、エージェントが自動編集しない
  （Hooksでも機械的にdeny）。変更が必要な場合は人間が直接編集する。
- 認証情報はコード/ドキュメントに直接書かない。保管場所（`environment.md` に記載の
  Databricks Secrets / GitHub Secrets等）への参照のみとする。
- CI/CDの実行主体にはservice principal + Workload Identity Federation（OIDC）を使い、
  長期PATを使わない。開発用IDにはprodへの書き込み権限を与えない（権限分離が最終ゲート）。
- 人の明示承認なしに production デプロイ・権限変更・データ削除・リリース承認を行わない。
- テストや安全対策を弱めてチェックを通さない。実行していない検証を成功と報告しない。

## コンテキスト管理（コンテキストロット対策）

長い会話ほどモデルの想起精度が落ちる（"context rot"）。このハーネスでは:

1. **ドキュメントが記憶であり、会話が記憶ではない。** `docs/` の各ファイル
   （`design-index.md`, `irr.md`, `tasks.md`, `test-report.md`, `progress.md` 等）が
   プロジェクトの正の状態。フェーズの切り替わりや会話が長くなったと感じたときは、
   新しいチャットセッションを始めて関連ドキュメントを読み直す方を優先してよい。
   実装〜テストの全自動区間でも「完了タスク10個超」「実装⇔テストの差し戻し2往復超」を
   目安に、区切りでセッションの切り替えをユーザーに提案する
   （具体的な分割基準の表はUSAGE.mdの「セッション運用の原則」）。
2. **サブエージェントによる実装ループの分離。** 1タスク=1 `task-worker` 呼び出し。
3. **独立レビューも同じ仕組み**（バイアス除去とコンテキストロット対策を兼ねる）。
4. セッションへ長い設計書を直接貼らない。設計IDと `design-index.md` 上のパス
   （または正式保管先）を指定する。

## コスト（トークン/AI Credit）についての方針

- 高頻度・機械的なエージェント（`orchestrator`, `implement`, `test`）は `model: auto`。
- 一度の判断ミスが高くつくエージェント（`design-gate`のIRR判定, `design-critic`,
  `reviewer`）は、利用可能な範囲で強いモデルへの切り替えを検討する
  （設計書チェックの見落としは実装全体の手戻りに直結する）。
- Skillの`description`は具体的でキーワードが立つように書く（段階的開示を機能させるため）。
- Hooksはモデルを介さないシェル実行でありほぼゼロコスト。「モデルに繰り返し指示して
  守らせる」より「Hooksで機械的に強制する」方が優れる場面では、Hooksを増やすことを検討する。

## 設計変更・改修・バグ修正（差分駆動の原則）

設計書の正はこのハーネスの外（設計工程）にある。改修は常に**差分駆動**で行い、
全設計書の再チェック・全タスクの見直しをしない（影響範囲は `traceability.md` から辿る）。

- **軽微な誤記**: 設計書の正式保管先で修正し、`design-index.md` の版を更新する。
- **実装に影響する設計変更・機能追加**: 設計書の修正は `/08-design-revise`
  （ハーネス内・人間合意つき）または設計工程（ハーネスの外）で行い、変更された
  設計IDについて `/02-design-check` を再実行してIRRの**該当項目だけ**を再判定する。
  承認済みIRRの前提が崩れたまま実装を続けない。タスクは該当設計ID分だけ
  `tasks.md` に追記する。
- **設計変更を伴わないバグ修正**（実装が設計とずれていた）: 設計書の修正もIRR再判定も
  不要。再現テストを先に追加し、最小修正で通す。
- **緊急ホットフィックス**（設計書更新を待てずコードを直接修正）: 許容されるが、
  再現テスト・PR・人の承認・記録は緊急でも省略しない。修正と同時に
  `design-index.md` の該当設計IDを「実装乖離あり（解消期限つき）」にし、
  `learnings.md` に1行追記する。期限までに設計書を更新して整合を回復する。
  乖離が未解消の設計IDは、以後のIRRで `CONDITIONAL GO` の条件として扱う。
  **黙って乖離したままにすることだけが禁止**である。
- `warn-stale-gate` フックが、承認済み文書の編集を検知して非ブロッキング警告を出すが、
  整合の確認は人（またはエージェントへの指示）の役目である。

## 成長ループ（このハーネスは使うたびに賢くなる設計）

1. **教訓（`docs/00-overview/learnings.md`）** — 次のいずれかに当たったら、
   その場で1行追記する（`- [YYYY-MM-DD] 教訓`）。SessionStartフックで全セッションに
   自動注入される。一般的なベストプラクティスはここではなくSkillに書く。
   - ユーザーに同じ趣旨の訂正を受けたとき
   - 同じアプローチで2回失敗したとき
   - プロジェクト固有の暗黙知が判明したとき
   - **コマンド・ツールの実行方法を試行錯誤の末に確立したとき**（2回以上失敗してから
     成功した実行手順・回避策は、**成功したコマンドをそのまま**記録する。
     例: `- [2026-07-05] devデプロイはCLIがPATHに無いため python -m databricks.sdk 経由で実行する`。
     仕様はdocsに残るが実行方法の獲得知識は書かないと消える。次のセッションが
     同じ試行錯誤を繰り返す最大の原因なので、仕様の教訓より優先して書く）
   - **セッションを終える・別フェーズへ引き継ぐ前に**、このセッションで確立した
     環境固有の実行方法が記録済みかを確認する（未記録のまま次へ渡さない）
2. **振り返り（`docs/05-retrospective/`）** — リリース後に `/07-retrospective` で実施。
3. **本体への還流** — 改善提案と再利用可能なSkillは、ハーネス本体リポジトリに人間が適用し、
   `DECISIONS.md` に根拠つきで記録する。
   エージェントの失敗が再発する場合はプロンプトを長くせず、コンテキスト・ツール・
   テスト・ガードレールの不足としてハーネスを改善する。

## 破壊的操作・外部影響のあるアクション

- `git push`・タグ付け・force系操作、`databricks bundle deploy/run` の staging/prod
  ターゲット指定、`bundle destroy`、`jobs delete`、スキーマ/テーブルの削除など、
  影響が大きい・後戻りしにくい操作は必ず事前にユーザーに確認してから実行する
  （Hooksでも機械的に確認が入るが、フックが発火しない環境でも必ず確認すること）。
- devターゲットへの `bundle deploy` / `bundle run` は全自動区間に含まれる（確認不要）。

## コーディング規約（実装フェーズ共通）

- 設計にない機能・抽象化を先回りして作らない（YAGNI）。
- コメントは「なぜ」を書くときだけ。「何をしているか」は書かない。
- テーブル名などの識別子とデータ値を区別し、SQL文字列の直接連結を避ける。
- 冪等性・再実行・部分失敗・遅延データ・タイムゾーン・NULLを常に設計IDの受け入れ条件と
  突き合わせて実装する。
- driverへの`collect`、無制限スキャン、不要なshuffleなど性能・コストに影響する
  パターンを避ける。
- テストは `docs/03-test/` のテストケースと `docs/02-implementation/traceability.md` に
  対応づけて書く。テストが実装の写経にならないよう、受け入れ条件から先に書く。

## ドキュメント規約

- `docs/` 配下の実体ファイルは対応する `*_template.md` の見出し構成を維持する。
- `*_template.md` 自体は編集しない（コピーして実体ファイルを作る）。
- 表形式の項目は空欄のまま残さず、「該当なし」等を明示する。
- 機密設計書はリポジトリへ複製せず、正式保管先と版・承認日を `design-index.md` に記録する。
