# 設計文書索引

## 目的

AIと開発者は最初にこの索引だけを確認し、登録された設計書だけを読む。
設計書本体のファイル名は固定しない（日本語・案件名を含む既存命名規則を利用できる）。
文書種別は配置ディレクトリで分け、追跡にはファイル名ではなく変更されにくい設計IDを使う。

```text
docs/01-design/
├── requirements/              # 要件文書がある場合だけ
├── basic-design/              # アーキテクチャ・外部設計
└── detailed-design/
    ├── features/              # 1機能1ファイルの詳細設計
    └── common/                # 共通機能ごとの詳細設計
```

## 文書一覧

| 設計ID | 種別 | 対象・機能 | ファイルまたは正式保管先 | 版・承認日 | 受け入れ条件ID | 状態 | 関連ID |
|---|---|---|---|---|---|---|---|
| REQ-001 / なし | 要件 | TODO | TODO | TODO | TODO | TODO | - |
| BD-001 | 基本設計 | 案件全体 | TODO | TODO | TODO | TODO | REQ-001 |
| DD-001 | 機能詳細 | TODO | TODO | TODO | TODO | TODO | BD-001 |
| DD-COM-001 | 共通詳細 | TODO | TODO | TODO | TODO | TODO | BD-001 |

状態: `登録済み` / `未承認` / `版確認中` など。`TODO`・未承認・版不明の文書を
実装根拠として使用しない。

## 登録規則

- 基本設計書は1ファイルでも複数ファイルでもよい。既存の組織命名規則を優先する。
- 詳細設計書は機能単位でファイルを分け、各行に別の設計IDを割り当てる。
- 機能詳細の行から受け入れ条件を辿れるようにする（設計IDだけで実装を開始できる状態）。
- 共通機能は `detailed-design/common/` へ分け、利用する機能詳細から関連IDで参照する。
- 要件文書がなければ要件行を「なし」とする。
- 外部文書管理基盤を正本とする場合は文書を複製せず、承認済み版へのリンクを登録する。
